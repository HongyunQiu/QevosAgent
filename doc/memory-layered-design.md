# Layered Memory Design

这份文档定义 `simpleAgent` 的下一版记忆系统设计，目标是把“任务级记忆沉淀”和“总体级记忆概述”拆成两层，既保留结构化真源，又避免主记忆退化成原始过程堆积。

## 设计目标

这次设计优先解决三个基础问题：

1. 任务结束时稳定生成一条结构化任务记忆
2. 从任务级记忆中周期性重写总体级 `memory.md`
3. 在预热阶段同时利用总体级记忆和少量相关任务记忆

重点先做基础功能，不在第一版引入崩溃恢复、复杂 embedding 检索或过多的人工分类标签。

## 总体结构

记忆系统拆成两层：

- `memory/task_memory.jsonl`
  - 任务级真源
  - 每个 run 一条极简记录
  - 面向后续重写、检索和增量处理
- `memory/memory.md`
  - 总体级概述
  - 不是条目拼接，而是对近期任务记忆按主题聚合后的自然语言总结
  - 面向预热阶段全量加载

这两层的职责不同：

- `task_memory.jsonl` 负责“不漏”
- `memory.md` 负责“不乱”
- 预热加载负责“不多”

## 模块边界

建议把记忆处理收敛到独立模块，例如 `agent/runtime/memory_engine.py`，只负责以下三件事：

1. `record_task_memory(...)`
  - 任务结束时生成或覆盖一条任务记忆
2. `maybe_refresh_main_memory(...)`
  - 判断是否需要重写 `memory.md`
3. `load_startup_memory(...)`
  - 预热阶段加载 `memory.md`
  - 并按需补充少量相关任务记忆

其他模块不再自行拼接或追加主记忆，只调用这个模块暴露出的接口。

## 任务级记忆：`task_memory.jsonl`

### 数据格式

每条记录只保留四个字段：

```json
{
  "run_id": "20260407-233451",
  "task": "全网搜索关于 Qwen3.5 27B 模型微调的方法",
  "summary": "检索并整理了 Qwen3.5 27B 的主流微调方法与框架，确认 LLaMA-Factory、Unsloth、Swift 等可用于该模型的微调实践。",
  "keywords": ["qwen3.5-27b", "finetune", "llama-factory", "unsloth", "swift"]
}
```

### 约束规则

- 同一 `run_id` 只保留一条记录
- 重复生成时覆盖旧记录，不追加第二条
- `task` 应是用户任务要求的压缩表达，不能直接写入完整冷启动前缀
- `summary` 必须是单段短摘要，不得退化成日志、分点清单或原始输出拼接
- `keywords` 建议控制在 `3-6` 个，只保留真正有助于检索的词
- 不引入 `kind`、`outcome` 等强分类或低信息密度字段

### 写入原则

每次任务结束都生成一条记录，不做“是否高价值”的前置筛选。

原因是：

- 任务价值无法在入口阶段稳定判断
- 真源层更应该追求完整性
- 后续是否上升到总体记忆，由 `memory.md` 重写阶段决定

### 生成素材来源

生成任务记忆时，优先使用运行后产物，而不是回读全量原始对话。建议读取优先级如下：

1. `execution_summary.md`
2. `final_answer.md`
3. `completion_review.json`
4. `reflection.md`

如果这些内容信息不足，再降级依赖更弱的运行上下文，但第一版不以全量 transcript 为主来源。

### 失败处理

任务记忆生成采用降级策略：

1. 优先生成完整四字段记录
2. 若失败，则降级生成一条最小可用记录
3. 若仍失败，只记录日志，不阻塞主任务结束

由于进程崩溃导致的缺失，第一版暂不解决，后续可由独立恢复机制补上。

## 总体级记忆：`memory.md`

### 定位

`memory.md` 是“阶段性总体记忆”，不是任务清单。

它的目标不是按 `run_id` 罗列历史，而是把近期反复出现的主题、能力演进和值得优先联想的方向，压缩成几段可直接用于预热的概述。

### 生成风格

`memory.md` 应满足以下要求：

- 按主题聚合，而不是按任务列举
- 使用自然语言短段落，不写成日志
- 强调“最近在做什么”“形成了什么能力或认识”
- 只保留对后续预热有帮助的趋势信息
- 不直接罗列工具清单、错误栈、完整中间产物

建议默认控制在 `3-6` 段短段落。

### 典型主题

`memory.md` 可以覆盖的主题包括：

- 近期 FPGA / 硬件开发工作
- 近期搜索 / 外部信息检索工作
- 近期 `simpleAgent` / 框架演化工作
- 可选的稳定状态或倾向变化

其中“状态/倾向型描述”允许出现，但必须谨慎：

- 只有在它已经表现为相对稳定趋势时才写入
- 不应把一次性情绪、偶发感受或短期波动直接写入长期总体记忆

### 示例风格

下面是一类符合预期的 `memory.md` 风格示例：

```md
最近进行了一些 FPGA 开发工作，包括 I2C 控制器开发、UART 串口开发，已经能实现从编写到编译再到仿真的整体闭环，为验证正确性提供了保障。

最近一周还进行了一些搜索工作，包括对 QWEN3.5 模型、GEMMA4 模型的搜索，发现了最新的一些进展，包括 4BIT 的 NVBF4，也发现了 Jang 实现了微调并发布了较受欢迎的微调模型。

近期还进行了 simpleAgent 的测试工作，产生了大量 runs 文件。simpleAgent 的代码在快速进化中，目前已经重写了预热架构，正在进行 memory 的新设计工作。

最近的工作热情很高，需要继续发扬。
```

在正式生成时，应保留这种“按主题概括”的风格，但对最后一类状态描述保持更严格的写入门槛。

## `memory.md` 的重写触发

`memory.md` 采用混合触发模式：按条件自动重写，也支持人工强制重写。

第一版建议以下任一条件满足时触发重写：

- `task_memory.jsonl` 新增了至少 `N` 条尚未纳入总体记忆的任务记录
- 近期任务出现了明显新主题，值得进入总体记忆
- `memory.md` 不存在、过短或明显过旧
- 人工显式触发重写

如果这些条件都不满足，则只更新 `task_memory.jsonl`，不改写 `memory.md`。

实现时可以使用一个轻量 sidecar 状态文件（例如 `memory/task_memory_state.json`）记录“上次重写 `memory.md` 时已纳入了多少条任务记忆”，用于避免当累计条目数超过阈值后每次任务结束都重复重写。

这样做的原因是：

- `task_memory.jsonl` 适合持续积累
- `memory.md` 更适合低频、稳定地演进
- 不是每个新任务都值得立刻改写总体记忆

## 预热阶段的记忆加载

预热阶段按两层加载：

### 第一层：总是加载 `memory.md`

这是总体背景层，负责提供：

- 近期主要工作方向
- 当前高频主题
- 应优先联想的长期背景

它适合全量进入预热上下文。

### 第二层：按需补充少量 `task_memory.jsonl`

这层不是全量加载，而是按当前任务检索出少量最相关记录，建议只补充 `3-5` 条。

它负责提供：

- 与当前任务最接近的历史任务摘要
- 可复用的近期经验线索
- 进一步定位历史 run 的入口词

### 第一版检索策略

第一版不引入 embedding，先采用轻量规则检索：

- 从当前任务文本切词
- 与 `task`、`summary`、`keywords` 做重叠匹配
- 计算简单相关分数
- 取最高分的前几条
- 再做主题去重，避免相似记录刷屏

这种方式实现简单，且足以支撑第一版预热补充。

## 为什么采用双层结构

如果只加载 `memory.md`，信息过于抽象，难以支撑具体联想。

如果只加载 `task_memory.jsonl`，信息过于碎片化，容易把预热上下文塞满。

双层结构可以同时满足：

- 总体方向感
- 具体相关性
- 上下文体积可控

因此，预热阶段建议默认组合使用：

- `memory.md` 作为总体背景
- 少量相关 `task_memory.jsonl` 作为补充细节

## 非目标

以下内容不纳入这一版：

- 崩溃后的任务记忆恢复机制
- 向量检索或 embedding 存储
- 多级分类体系
- 复杂的人工标签管理
- 过多的运行态统计字段

这一版只聚焦把“任务级记忆写入”“总体记忆重写”“预热双层加载”三条基础链路做稳。

## 最终结论

这套设计的核心是：

- 用 `task_memory.jsonl` 保存完整、极简、结构化的任务级真源
- 用 `memory.md` 保存按主题聚合后的阶段性总体记忆
- 在预热阶段同时加载总体记忆和少量相关任务记忆

这样既能避免主记忆退化成历史堆砌，又为后续更高层级的“历史描述重写”保留了清晰的演化空间。
