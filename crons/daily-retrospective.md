# 每日复盘任务

现在的时间是凌晨 3:00，请对**昨天**（按本地时区计算）的工作做一次复盘。

## 步骤

1. **定位昨天的 runs**
   - 看 `runs/` 目录下日期前缀（形如 `YYYYMMDD-HHMMSS`）属于昨天的所有子目录。
   - 如果昨天没有任何 run，直接在报告里写"昨日无任务"并跳到第 4 步。

2. **逐个 run 做轻量提取**（不要逐字读全部日志，按需采样即可）
   - 读每个 run 的 `status.json`、`scratchpad.md`、`meta.json` 拿到任务目标、最终状态、关键产物。
   - 必要时翻 `short_term.jsonl` 的最后若干条，定位 done / error 事件。

3. **汇总分析**，至少回答下面这些：
   - 昨天一共跑了多少次任务？成功 / 失败 / 中断 各多少？
   - 失败或中断的任务，根因是什么？是工具问题、提示词不清晰，还是外部依赖？
   - 有哪些**重复出现**的坑（同类报错、同类绕路）？
   - 如果让你今天重做这些任务，**有哪些经验可以提前注入到工作流里**？

4. **写报告**
   - 输出到 `reports/retro-YYYY-MM-DD.md`（文件名里的日期是**昨天**的日期）。
   - 报告结构：
     ```
     # 每日复盘 — YYYY-MM-DD

     ## 任务概况
     ## 成功亮点
     ## 失败与中断
     ## 重复出现的模式
     ## 经验教训（写给未来的自己）
     ```

5. **沉淀长期记忆（重要）**
   - 把"经验教训"中**真正可复用、非昨天专属**的内容，追加到 `memory_macro.md`，作为今后所有任务的背景知识。
   - 不要照搬整段，挑出 1–3 条最有价值的，写成简短规则或反模式。
   - 写之前先读一遍 `memory_macro.md`，避免重复或矛盾。

## 注意

- 全程不需要任何外部网络访问。
- 报告写完后请 `done`，不要进入持续对话模式。
- 如果 `runs/` 为空或不存在，照常生成一份"无任务"报告，并 `done`。
- 全程不允许使用ask_user工具，因为用户在睡觉无法响应。
