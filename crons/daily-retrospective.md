# 每日复盘任务

现在的时间是凌晨 3:00，请对**昨天**（按本地时区计算）的工作做一次复盘。

## 步骤

1. **定位昨天的 runs**
   - 看 `runs/` 目录下日期前缀（形如 `YYYYMMDD-HHMMSS`）属于昨天的所有子目录。
   - 如果昨天没有任何 run，直接在报告里写"昨日无任务"并跳到第 4 步。

2. **逐个 run 做轻量提取**（不要逐字读全部日志，按需采样即可）
   - 读每个 run 的 `status.json`、`scratchpad.md`、`meta.json` 拿到任务目标、最终状态、关键产物。
   - 必要时翻 `short_term.jsonl` 的最后若干条，定位 done / error 事件。
   - **重点：读 `meta.json` 的 `_user_injections` 字段**——这是用户在运行途中的人工干预记录。
     每条含 `iter`（发生的迭代号）、`content`（用户说了什么 = 纠正后的"优动作"）、
     `agent_pending_action`（被打断时 Agent 正要做/刚做的动作 = 被否决的"劣动作"）。
     **人工干预是信噪比最高的信号，务必逐条看。**

3. **汇总分析**，至少回答下面这些：
   - 昨天一共跑了多少次任务？成功 / 失败 / 中断 各多少？
   - **【最高优先级】人工干预分析**——逐条过每个 run 的 `_user_injections`：
     - 用户为什么在那个点出手？对照 `agent_pending_action`（Agent 原本要做的"劣动作"）
       和 `content`（用户纠正成的"优动作"），提炼出这次干预背后的**意图**。
     - 判定这条干预属于哪一类：
       - **可泛化的通用规则**（如"这类设备必须先复位再连接"）→ 值得沉淀；
       - **一次性上下文**（如"今天这台机器临时坏了，手动重连一下"）→ 不该沉淀。
     - 同一类干预是否在**多个 run 里反复出现**？跨 run 复现 = 强沉淀信号。
   - 失败或中断的任务，根因是什么？是工具问题、提示词不清晰，还是外部依赖？
   - 有哪些**重复出现**的坑（同类报错、同类绕路）？
   - 如果让你今天重做这些任务，**有哪些经验可以提前注入到工作流里**？

4. **写报告**
   - 输出到 `reports/retro-YYYY-MM-DD.md`（文件名里的日期是**昨天**的日期）。
   - 报告结构：
     ```
     # 每日复盘 — YYYY-MM-DD

     ## 任务概况
     ## 人工干预复盘（逐条：意图 / 优劣动作对照 / 通用规则 or 一次性 / 是否跨 run 复现）
     ## 成功亮点
     ## 失败与中断
     ## 重复出现的模式
     ## 经验教训（写给未来的自己）
     ```

5. **沉淀长期记忆（重要，自动执行，无需征询用户）**
   - **优先**把上面"人工干预分析"中判定为**可泛化通用规则**的干预，提炼成简短规则，
     追加到 `memory_macro.md`，作为今后所有任务的背景知识。
   - 其次再补充其它"经验教训"中**真正可复用、非昨天专属**的内容。
   - 判定为**一次性上下文**的干预**绝不要**写入，避免把偶发噪声固化进未来所有 run。
   - 不要照搬整段，挑出最有价值的若干条，写成简短规则或反模式。
   - 写之前先读一遍 `memory_macro.md`，避免重复或矛盾。

## 注意

- 全程不需要任何外部网络访问。
- 报告写完后请 `done`，不要进入持续对话模式。
- 如果 `runs/` 为空或不存在，照常生成一份"无任务"报告，并 `done`。
- 全程不允许使用ask_user工具，因为用户在睡觉无法响应。
